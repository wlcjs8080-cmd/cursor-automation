# DB 마이그레이션 계획 (2026-03-05 확정)

## 목표
마스터 시트와 스케줄 시트의 동시 접근 충돌을 해결하기 위해 SQLite DB를 도입한다.
4명이 동시에 자동화를 실행해도 기다림 없이, 충돌 없이 작동하는 구조를 만든다.

## 현재 문제점
- xlwings로 엑셀 파일을 열면 파일 잠금(lock)이 걸려 다른 사람이 동시에 사용 불가
- 마스터 시트 1개를 여러 명이 동시에 읽기/쓰기 불가능
- 스케줄 시트도 자동화 실행 시 엑셀을 열어야 하므로 동시 실행 불가

## 해결 구조: SQLite DB 도입

### 파일 구성
- master.db: 마스터 시트 데이터 (영구 보관, 계속 누적)
- 스케줄 DB: 스케줄 시트 데이터 (자동화 시작 시 최신 내용 반영, 덮어쓰기 아닌 추가/업데이트 방식)
- 레포트 파일: 기존과 동일하게 엑셀 파일로 유지 (각자 독립적)

### 전체 업무 흐름
1. 사람이 SharePoint 스케줄 시트에 입력 (기존과 동일)
2. 자동화 시작 버튼 → 스케줄 엑셀을 읽어서 스케줄 DB에 추가/업데이트
3. Step 1: 스케줄 DB에서 미완료 행을 읽고, 레포트 템플릿 복사하여 생성, 스케줄 DB에 "완료" 표시
4. Step 2: 각자 레포트 열고 품번 입력 후 실행 → 마스터 DB에서 이전 교체일 조회 (읽기만)
5. Step 3: 레포트 + 스케줄 DB 데이터를 읽어서 마스터 DB에 새 행 기록 (쓰기)
6. 퇴근 시: 내보내기 버튼 → 마스터 DB 전체를 마스터 엑셀로 저장, 스케줄 DB "완료"를 원본 스케줄 엑셀에 반영

### SQLite 동시 접근 특성
- 동시 읽기: 제한 없음 (몇 명이든 동시에 읽기 가능)
- 동시 쓰기: SQLite가 자동으로 순서 관리 (0.1초 대기 후 순차 처리, 데이터 유실 없음)
- 파일 잠금: 엑셀처럼 전체 파일을 잠그지 않음

### DB 초기 설정 (최초 1회)
- 기존 마스터 엑셀의 전체 데이터를 master.db로 변환하는 스크립트 실행
- 이후 Step 3에서 새 데이터가 자동 누적됨

### 코드 변경 범위
- 변경됨: 마스터 시트를 읽고 쓰는 함수들 (xlwings → SQLite)
- 변경됨: 스케줄 시트를 읽는 함수들 (xlwings → SQLite, Step 1 시작 시 DB 생성)
- 변경 없음: 레포트 템플릿 복사 (shutil.copy2 그대로)
- 변경 없음: 레포트 셀 채우기 (xlwings 그대로)
- 변경 없음: 레포트 읽기 (xlwings 그대로, Step 2/3에서 각자 레포트 열기)
- 추가: DB 초기화 스크립트 (마스터 엑셀 → DB 변환)
- 추가: 내보내기 스크립트 (DB → 마스터 엑셀, DB → 스케줄 엑셀 완료 반영)

### 스케줄 DB 관리 규칙
- 덮어쓰기 금지: 새로 자동화를 시작해도 기존 "완료" 표시를 유지
- 추가/업데이트 방식: 새 행은 추가, 기존 행은 상태만 업데이트
- 이전 사람이 처리한 "완료"는 보호됨

### 마스터 DB 관리 규칙
- 데이터 영구 보관 (삭제 없음, 계속 누적)
- 내보내기는 DB의 복사본을 만드는 것이며 DB를 비우지 않음
- 다음날에도 전날 데이터가 그대로 유지됨

### 안정성 검증 결과 (시나리오 테스트 완료)
- 4명이 서로 다른 시간에 자동화를 실행해도 충돌 없음 확인
- 동시에 마스터 DB에 쓰기를 해도 SQLite가 순차 처리하여 안전
- 한 명이 완료하기 전에 다른 사람이 시작해도 각자 DB를 통해 독립적으로 작동
- 유일한 규칙: 퇴근 시 한 명이 내보내기 실행

### 구현 우선순위
1. Step 1~3 통합 테스트 완료 (현재 진행 중)
2. DB 초기화 스크립트 작성 (마스터 엑셀 → master.db)
3. Step 1 코드 수정 (스케줄 DB 생성 + DB에서 읽기)
4. Step 2 코드 수정 (마스터 DB에서 읽기)
5. Step 3 코드 수정 (스케줄 DB 읽기 + 마스터 DB 쓰기)
6. 내보내기 스크립트 작성 (DB → 엑셀)
7. 전체 통합 테스트 (4명 동시 실행 시나리오)
8. .exe 앱 변환 (GUI에 자동화 버튼 + 내보내기 버튼)

### 배경: 비교 검토 완료
- 클라우드 전면 전환 (Office Scripts + Power Automate): TypeScript 전면 재작성 필요, 3~4주, 비추천
- 마스터 5개 분리 방식: 안정적이나 합치기 복잡, 용량 5배, 관리 부담
- SQLite DB 방식: 기존 코드 최소 수정, 동시 접근 안전, 기다림 제로 → 채택


## DB 마이그레이션 핵심 정리 (2026-03-05 확정, 시뮬레이션 검증 완료)

### 구조
- 스케줄 엑셀: 읽기 전용으로만 접근 → SQLite DB(스케줄 DB)로 복사
- 마스터 시트: SQLite DB(master.db)로 전환
- 레포트 파일: 기존 엑셀 유지 (변경 없음)

### Step 1 안전장치
1. 처리할 행을 읽는 즉시 DB에 "처리중" 표시 (SELECT + UPDATE를 하나의 트랜잭션으로 묶어서 동시 중복 차단)
2. 레포트 생성 전 고객사/REPORT 폴더에 같은 파일명이 이미 존재하면 생성하지 않고 "같은 파일명의 레포트가 존재합니다" 경고 메시지 출력
3. 에러로 중단 시: 생성된 레포트 파일 삭제 → DB 상태 "미완료"로 되돌리기 (이 순서 필수)
4. 에러 시 사용자 메시지: "[에러] 행 OOO (고객사: OOO, S/N: OOO) 레포트 생성 중지. 스케줄 시트 작성 내용은 유지됨. 문제 확인 후 Step 1 재실행 필요."
5. 정상 완료 시: "처리중" → "완료"로 변경

### Step 3 안전장치
- 한 건의 작업행(1개) + 파트행(여러 개)을 하나의 트랜잭션으로 묶어서 INSERT (행 순서 뒤섞힘 방지)

### 동시 접근 안전성 (8명 시뮬레이션 검증 완료)
- 스케줄 엑셀: 읽기 전용이라 몇 명이 동시에 열어도 충돌 없음
- 스케줄/마스터 DB: SQLite가 동시 쓰기 자동 순차 처리
- 레포트 파일: 각자 다른 파일이라 간섭 없음
- 같은 S/N이라도 방문 날짜가 다르면 파일명이 달라서 충돌 없음
- 같은 날짜 + 같은 S/N 중복 등록은 실무상 없음 (만약 실수로 발생 시 파일명 중복 체크에서 차단됨)

### 퇴근 시 내보내기
- 한 명이 내보내기 실행
- 마스터 DB 전체를 마스터 엑셀로 저장
- 스케줄 DB "완료"를 원본 스케줄 엑셀에 반영


## SQLite 네트워크 제한사항 (2026-03-05 리서치 확인)

### 문제 발견
- SQLite 공식 문서(sqlite.org/useovernet.html)에서 네트워크 공유 폴더 사용을 경고
- WAL 모드는 네트워크 드라이브에서 작동하지 않음 (로컬 파일시스템 전용)
- 네트워크에서 WAL 모드 사용 시 DB 파일 손상(corruption) 위험

### 해결: rollback 모드 사용
- SQLite 공식 문서에서 rollback 모드는 네트워크에서도 작동한다고 명시
- WAL 모드 대신 rollback 모드(journal_mode=DELETE)를 사용
- DB 파일은 기존 계획대로 서버 공유 폴더에 위치

### rollback 모드 동작 방식
- 동시 읽기: 여러 명 가능
- 동시 쓰기: 한 명씩 순차 처리 (SQLite가 자동 관리)
- 쓰기 중 다른 사람이 읽기/쓰기 시도: busy_timeout 시간 동안 자동 대기 후 재시도
- 쓰기 1건 소요 시간: 0.01~0.05초
- 8명이 정확히 동시에 실행해도 순차 처리가 0.1~0.4초 안에 완료
- 사람은 대기를 체감할 수 없음. 다시 누를 필요 없음.

### DB 연결 시 필수 설정 (모든 Step 공통)
- journal_mode=DELETE (rollback 모드, WAL 절대 금지)
- busy_timeout=10000 (10초, 잠금 대기 시간)
- 10초 안에 잠금이 안 풀리면 "DB가 사용 중입니다. 잠시 후 재실행 필요." 메시지 출력

### 기존 계획과 달라진 점
- 변경: WAL 모드 → rollback 모드(journal_mode=DELETE)
- 변경: busy_timeout 설정 필수 추가
- 유지: DB 파일 위치는 서버 공유 폴더 그대로
- 유지: Step 1 안전장치 (처리중 상태, 파일명 중복 체크, 에러 시 되돌리기) 전부 그대로
- 유지: Step 3 안전장치 (트랜잭션 묶기) 그대로
- 유지: 동시 접근 안전성 (8명 시뮬레이션 결과) 그대로
- 유지: 퇴근 시 내보내기 방식 그대로
- 유지: 구현 우선순위 순서 그대로

## DB 안정성 보완 (2026-03-05 추가)

### 자동 백업
- 매 실행 시 master.db → master_backup_YYYYMMDD.db 자동 백업 생성
- 퇴근 시 DB → Excel 내보내기가 최종 백업 역할

### 무결성 검사
- 매 실행 시 PRAGMA integrity_check 수행
- 실패 시 "DB 무결성 검사 실패. 백업 DB를 확인하세요." 메시지 출력 후 자동화 중단

### 파일명 중복 체크 (Step 1)
- 레포트 생성 전 고객사/REPORT 폴더에 동일 파일명 존재 여부 확인
- 존재 시 레포트 생성하지 않고 스킵 + 메시지: "같은 파일명의 레포트가 존재합니다"
- DB 상태를 "완료"로 바꾸지 않음

### 에러 메시지 형식 (Step 1)
- 확정 문구: "[에러] 행 OOO (고객사: OOO, S/N: OOO) 레포트 생성 중지. 스케줄 시트 작성 내용은 유지됨. 문제 확인 후 Step 1 재실행 필요."

### 동시 사용 검증 결과 (8명 시뮬레이션)
- Step 1 중복 처리: "처리중" 상태로 해결
- Step 3 행 순서: 단일 트랜잭션 INSERT로 해결
- 같은 레포트 동시 편집: 운용 규칙으로 해결 (각자 담당 레포트만 작업)
- Step 2: 마스터 DB 읽기 전용, 충돌 없음
