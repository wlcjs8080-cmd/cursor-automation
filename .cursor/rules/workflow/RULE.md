# 세션 시작 절차

사용자가 새 작업 세션을 시작하면 에이전트는 아래 순서를 따르라:

1. .cursor/rules/ 안의 3개 RULE.md를 모두 읽는다 (alwaysApply: true이므로 자동 로드됨)
2. project-context/RULE.md의 "현재 상태"와 "다음 작업 목록"을 확인한다
3. 사용자가 "현재 상태 요약해줘" 또는 "다음 작업 뭐야"라고 물으면 즉시 답변한다
4. 사용자가 작업 지시를 하면, 다음 작업 목록의 우선순위를 참고하여 진행한다


# 세션 종료 절차

사용자가 "오늘 작업 정리해줘"라고 하면:

1. 이번 세션에서 완료한 작업을 요약한다
2. project-context/RULE.md의 "세션 히스토리"에 새 세션 기록을 추가한다
3. project-context/RULE.md의 "현재 상태"를 최신으로 업데이트한다
4. project-context/RULE.md의 "다음 작업 목록"에서 완료 항목에 ✅ 표시한다
5. 작업 중 발견된 코딩 교훈이 있으면 coding-standards/RULE.md에 추가한다
6. 작업 중 발견된 절차 규칙이 있으면 이 파일(workflow/RULE.md)에 추가한다


# RULE.md 업데이트 규칙

에이전트가 RULE.md 파일을 수정할 때 반드시 지킬 것:

- 기존 내용을 절대 삭제하지 마라. 추가만 하라.
- 한 번에 한 파일만 수정하라. 동시에 여러 RULE.md를 수정하지 마라.
- 수정 전에 해당 파일의 현재 내용을 먼저 읽어서 구조를 파악하라.
- 수정 후 사용자에게 "X 파일의 Y 섹션에 Z 내용을 추가했습니다"라고 보고하라.
- 사용자가 "알맞은 RULE.md에 추가해줘"라고 하면:
  1. 3개 RULE.md의 역할 설명을 비교한다
  2. 가장 적합한 파일을 판단한다
  3. 해당 파일의 맞는 섹션에 추가한다
  4. 어디에 추가했는지 사용자에게 알려준다


# 에이전트 행동 제한 규칙

- 웹 검색하지 마라
- 파일을 삭제하지 마라
- 지시한 것만 하라
- 한 번에 하나만 수정하라
- 사용자가 승인하지 않은 작업을 임의로 하지 마라
- 하겠다는 작업 내용을 먼저 사용자에게 보고하고, 확인 후 진행하라


# 파일 수정 안전 규칙

- 코드 수정 시 수정 범위를 최소화하라
- 함수 단위로 수정하고, 수정하지 않는 함수는 건드리지 마라
- 대규모 리팩토링은 사용자 승인 후에만 진행하라
- 수정 전에 어떤 함수/파일을 수정할 것인지 먼저 알려라
- Step 진행 후 반드시 .cursor/rules/ RULE.md 업데이트 + 사용자에게 보고


# 테스트/검증 절차

코드 수정 후:
1. 터미널에서 python 파일명.py로 실행
2. 에러 없이 완료되는지 확인
3. 실제 엑셀 파일에서 결과 확인
4. 테스트 통과 후 사용자에게 보고

Step 2 테스트:
1. 레포트 파일을 열고 I57~I63에 품번 수동 입력
2. 앱에서 Step 2 버튼 클릭
3. K57~K63에 이전 교체일이 정상 입력되는지 확인


# 터미널 명령어 안내 규칙

- 클릭으로 할 수 있는 건 클릭하라고 안내하라 (터미널 강요 금지)
- 여러 줄 명령어는 "한 줄씩 엔터"인지 "한번에 복붙 가능"인지 반드시 표시하라
- PowerShell 환경임을 항상 고려하라
- 명령어가 3줄 이상이면 번호를 매겨라
- 위험한 명령어(삭제, 덮어쓰기)는 실행 전 경고하라


# 사용자와의 소통 규칙

- DK님은 코딩 초보다. 전문용어 쓸 때는 쉬운 설명을 함께 붙여라.
- 코드 변경 사항을 설명할 때 "왜 바꾸는지"를 먼저 말하고 "어떻게 바꾸는지"를 말하라.
- 한 번에 여러 파일을 동시에 수정하지 마라. 한 파일씩 수정하고 확인받아라.
- 에러가 발생하면 에러 메시지의 핵심을 한국어로 번역해서 설명하라.
- "이거 왜 안 돼?" 같은 질문에는 원인을 먼저 말하고, 해결 방법을 바로 제시하라.
